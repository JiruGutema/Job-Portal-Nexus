GitHub Collaboration Rules & Agreements

1. Branching Strategy

  • Feature branches → Create a new branch per feature or bug fix:

    • Naming: feature/job-crud, bugfix/login-error, hotfix/security-patch.

  • Always update your branch with the latest dev before making a PR.

2. Commit Message Guidelines

Follow the Conventional Commits style:

<type>(<scope>): <short description>

Types:

  • feat → New feature
  • fix → Bug fix
  • docs → Documentation changes
  • style → Code style changes (formatting, linting)
  • refactor → Code restructuring without new features/bugs
  • test → Adding or updating tests
  • chore → Maintenance tasks

Examples:

  • feat(auth): add JWT login endpoint
  • fix(jobs): correct pagination bug
  • docs(api): update job application flow

3. Pull Requests (PR)

  • Every change must go through a Pull Request into dev.

  • Title should describe the change: Add job application API.

  • PR must include:

    • Clear description of the changes.
    • Linked issue number (if any).
    • Screenshots for UI changes (if applicable).

  • At least 1 reviewer approval required before merging.

4. Code Style

  • Follow agreed linting rules (e.g., ESLint/Prettier for Node.js, PEP8 for Python).
  • Write clean, modular, and well-documented code.
  • Keep functions small and focused.

5. Testing

  • Write unit tests for new features.
  • Ensure all tests pass before opening a PR.
  • Run npm test / pytest (depending on stack) before committing.

6. Issues & Task Management

  • Use GitHub Issues to track bugs, features, and tasks.
  • Use labels (bug, enhancement, documentation, urgent) for clarity.
  • Assign yourself before starting an issue to avoid conflicts.

7. Collaboration & Communication

  • Use GitHub Discussions / Slack / Teams (as agreed) for team communication.
  • Respect review feedback — collaboration is about improvement, not criticism.
  • Always document big architectural or design changes in the README or Wiki.

8. Do’s & Don’ts

✅ Do:

  • Commit often, push regularly.
  • Write meaningful commit messages.
  • Keep PRs small and focused.
  • Review others’ code respectfully.

❌ Don’t:

  • Commit directly to main.
  • Push broken or untested code.
  • Leave commented-out / debug code in PRs.
